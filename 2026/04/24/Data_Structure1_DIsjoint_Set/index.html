Data_Structure 1: Disjoint Set(Union Find)

From this article, I will introduce some easy data structures in that category. The first one is Union Find.

Why we develop this data structure

The genesis of Union-Find stems from the need to solve the Equivalence Problem in computer science. In early computational theory, researchers were looking for an efficient way to process a series of “equivalence relations” (e.g., a≡ba \equiv ba≡b, b≡cb \equiv cb≡c) and determine if two elements belong to the same equivalence class.

In the classic paper “An Improved Equivalence Algorithm,”[1] the core motivation was to move beyond simple linked lists or arrays, which were computationally expensive for large datasets. Before Union-Find, checking if two elements were connected often required traversing the entire structure, leading to O(N)O(N)O(N) time complexity for a single query.

The Union-Find Data Structure

The Union-Find data structure, also known as Disjoint Set Union (DSU), is designed to efficiently manage a collection of disjoint sets. It provides two primary operations: Find and Union.

We get the idea from the Algorithm 2 to use a set of trees(or forests) to represent the sets. Each tree represents a set, and the root of the tree is the representative (or “leader”) of that set. Each node in the tree points to its parent, and the root node points to itself.

Path Compression

One optimization technique for Union-Find is path compression which have appeared in Algorithm 3. During a Find operation, we can make every node on the path from the queried node to the root point directly to the root. This significantly speeds up future queries.

Union by Rank

Another optimization is union by rank. If we simply perform a union by making one root point to another, we might accidentally create a very deep tree (in the worst case, a degenerate tree that looks like a linked list).

Instead of randomly picking which tree becomes the parent, we track a “rank” for each node. Initially, every node starts with a rank of 0. When it comes time to merge two sets:

1.We compare the ranks of the two roots.

2.The root with the lower rank is attached under the root with the higher rank.

3.If both roots have the same rank, we pick one to be the parent and increment its rank by 1.

The reason why we don’t maintain the height is that once we apply Path Compression, the actual height of the tree changes constantly. “Rank” serves as an upper bound on the height of the tree and remains a stable metric for making merging decisions.
